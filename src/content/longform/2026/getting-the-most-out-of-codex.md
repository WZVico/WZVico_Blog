多数开发者初次接触编程智能体，往往是用来处理纯代码工作：审查代码库、生成差异对比、运行测试，然后提交拉取请求。

代码处理固然仍是 Codex 的拿手好戏。但在电脑上，大量日常工作实则都由代码在幕后驱动：执行 Shell 命令、浏览网页、调用 API、导出文档、响应事件以及触发自动化流程。随着 Codex 逐渐接管这些应用层面，它给人的感觉不再局限于狭义上的编程助手，而更像是一个能帮你搞定各种电脑操作的全能系统。

[Codex app](https://developers.openai.com/codex/app/features/) 将这种转变落到了实处。一个对话线程不仅能记住上下文、调用工具、展示生成制品，还能在多次提示词之间连贯推进，而不会在每次问答后就重置清空。

要想彻底释放 Codex 的潜能，关键在于打通并组合运用以下能力：

- 能够保留上下文的持久化线程
- 在用户深度参与时，运用语音输入、任务干预与排队机制
- 借助浏览器、电脑操控、MCP 服务器及各种连接器，让 Codex 的能力跃出代码库的边界
- 利用线程自动化与目标驱动，即使用户离开也能让工作继续运转
- 善用侧边栏，轻松审阅代码、文档、幻灯片及其他产出物

## **持久化线程**

> **持久化线程：** 长期运行的 Codex 线程，能够在多次会话中完整保留工作上下文。

置顶线程是让持久化线程触手可及的好方法。这对于处理以下高频重复的工作流极为管用：

- 充当“幕僚长”的专属线程
- 版本发布管理线程
- 文档审查线程
- 专门用于外部监控的线程

这些不仅仅是短暂的闲聊，而是持久的工作空间。Codex 能够随时回溯这些线程，保留过往的决策、偏好设定以及工作背景，省去了每次都要从头搭建上下文的麻烦。

置顶线程快捷键让这一操作变得非常实用。按下 Command-1 到 Command-9，即可一键跳转至保存好的线程。

## **语音输入**

语音输入的价值在于，它能在思想被精雕细琢成文字之前，捕捉到最原始、最粗糙的灵感火花。

Codex 内置了语音输入功能。当你只有一个模糊的起点，用嘴说很顺畅、用手敲字却很别扭时，这招尤为奏效：

我记得 Slack 里好像有个叫 Ben 的提过这事。
具体细节记不清了。
你去查一下。

对于一个懂得自行搜索、收集上下文并汇报的智能体来说，这点提示往往就足够了。

在任务尚未完全成型时，用语音随口倾倒个两三分钟的思绪，效果同样极佳。

会议记录也是同样的道理。一份原汁原味的会议记录或口述的规划笔记，往往比一份简短的总结更适合作为素材，因为它保留了原本的不确定性、语气重音以及那些未尽的思路。

## **干预与排队**

当语音输入与对运行中任务的精准控制结合时，便会爆发更大的威力。

> **干预 (Steering)：** 在当前步骤完成之前，打断 Codex 正在执行的任务并赋予新的方向。

当智能体跑偏，需要及时纠偏时，干预操作就能派上大用场。例如在审查网站时，用户可以在侧边栏对界面进行批注，顺势打断当前工作：

- 把这个调小点
- 这两个元素之间的间距看着不太对劲
- 这段文案写错了

> **排队 (Queuing)：** 在当前步骤完成后，为 Codex 安排后续任务。

排队则截然不同。它不会打断正在进行的任务，而是将新任务加进队列。比如你可以这样吩咐：

工作完成后，把预览链接发给 Slack 里的审核人。

干预改变的是 Codex 此刻的操作。排队决定的是 Codex 接下来要做什么。两者都能让用户在任务推进的过程中始终把控大局。

## **工具与触达范围**

一旦线程具备了连续性，接下来的问题就是它能对什么施加影响。Codex 的触角可以层层向外延伸：

- [$browser](https://developers.openai.com/codex/app/browser) 用于侧边栏的应用内浏览器，Codex 可在此审查并批注网页界面
- [@chrome](https://developers.openai.com/codex/app/chrome-extension) 用于接管已登录的浏览器状态，处理基于 Chrome 的工作流
- [@computer](https://developers.openai.com/codex/app/computer-use) 用于处理只能依靠桌面 GUI 完成的工作

[$browser](https://x.com/search?q=%24browser&src=cashtag_click) 完美契合侧边栏的浏览器审查工作。[@chrome](https://x.com/@chrome) 专攻需要依赖用户 Chrome 登录状态的任务。[@computer](https://x.com/@computer) 则是搞定纯桌面 GUI 任务的利器。

MCP 服务器与各种连接器将这一理念拓展到了工作流的其余环节。[Slack](https://developers.openai.com/codex/integrations/slack)、[Gmail](https://developers.openai.com/api/docs/guides/tools-connectors-mcp) 和 [Calendar](https://developers.openai.com/api/docs/guides/tools-connectors-mcp) 等集成之所以关键，是因为许多重要任务在蜕变成代码之前，最先表现为一条消息、一封邮件或是一个排期难题。

技能让重复性工作流变得可复用。一旦摸索出某个好用的工作流，不妨[将其打包成一项技能](https://developers.openai.com/codex/skills)，这样 Codex 就能随取随用，省得每次都得从头学起。

## **无处不在的办公体验**

[Codex 移动应用](https://openai.com/index/work-with-codex-from-anywhere/)彻底打破了“必须坐在工位上”的死板规定。一项任务完全可以在 Mac 上启动——这里不仅有现成的文件、权限，还有配置好的本地环境——随后，哪怕你离开工位只用手机随时查看进度，任务照样能顺利推进。

这种体验在碎片时间尤为奏效。你可以把长耗时任务甩给 Codex 然后离开电脑前，在外面用手机答复它的疑问、批准下一步操作，或者在赶回来之前重新调整任务方向。本地环境稳如泰山，而你早就实现了人身自由。

## **自动化流程**

[自动化流程](https://developers.openai.com/codex/app/automations)能让 Codex 按设定的日程运转。如果某个重复性任务每次都需要一个崭新的工作空间（比如生成每日报告或定期检查代码库），定时自动化是绝佳选择。如果希望任务在既定的日程表下，带着上下文继续回到活跃的对话中，那么线程自动化便不可或缺。

> **线程自动化：** 心跳式的定时唤醒机制，按计划准时回到同一个 Codex 线程继续工作。

置顶线程确实好用，但它仍需被动等待用户回归。线程自动化则能每隔几分钟或几个小时主动巡视一圈，一直跑到满足设定条件为止，还能随着时间推移灵活调整检查频率。

比如设定一个“幕僚长”线程每 30 分钟跑一次：

每 30 分钟，检查一下 Slack 和 Gmail 里需要我关注但还没回复的消息。
帮我排个优先级，挑出最要紧的。
如果有人向我提问，尽可能深入调查并替我起草一份回复，但先别发送。

等你忙完回来，最耗时费力的上下文收集工作早就搞定了。至于发不发、怎么发，拍板权依旧在你手里。

线程自动化也是处理反馈循环的一把好手。它可以盯紧拉取请求评论、Google Docs 批注或 Slack 回复，趁你不在的时候继续推进周边工作。

想象一下这样一个动画制作流：审核人在 Slack 里分享了一段视频。线程自动化会定时检查这个线程，一旦有新意见飘进来，立马渲染更新版本，并在原帖下打标签回复审核人。要是集成工具搞不定最终上传这一步，桌面自动化还能接管 GUI 妥妥收尾。

整个大循环跨越了用于反馈的 Slack、用于渲染的代码库，以及负责最终上传的桌面自动化，行云流水。

## **目标驱动**

只有当任务设定了明确的终点线，让智能体有方向可冲刺时，目标驱动的威力才能发挥到极致。一个毫无力度的目标就像这样：

> **目标 (Goals)：** 耗时较长的 Codex 任务，设有一条明确的终点线，智能体会随着时间推移不断向其推进。

把这个 Markdown 文件里的计划实现一下。

强有力的目标应该具备可量化的成功标准。

举个例子，工程师要把一个内部工具从 Python 迁移到 Rust，完全可以先建好新目录，明确目标并立下死规矩：除非单元测试全部跑通，否则新版本的迁移绝不罢休。

一个好目标，是持续执行力与验证器的完美结合。用户只需定好结果、给出停止条件，并提供判断 Codex 是否离终点更近的明确信号。

实用的验证器包括：

- 测试套件
- 基准测试
- Bug 复现机制
- 验证矩阵
- 必须始终通过的端到端工作流

雄心壮志固然重要，但少了验证机制，就沦为空谈了。

## **侧边栏**

[侧边栏](https://developers.openai.com/codex/app/features)巧妙地将工作成果与催生它的对话并列呈现。你不必再导出文件、在不同窗口间疲于奔命，直接就能原地审阅。产出物可能是代码，但也极可能是幻灯片、PDF、网页、表格或是中途生成的任何制品。

它在处理这四项工作时尤为得心应手：

- 检查生成制品
- 批注修改意见
- 操作网页界面
- 审阅变更情况

有了侧边栏，用户就能原地查阅 Markdown、电子表格、数据表、文档及幻灯片。检查、做标记、大修大改，一切一气呵成，工作流绝不掉链子。

批注功能

幻灯片或 PDF 稳稳停在生成它的对话旁，随时等你直接审阅和修改。

[应用内浏览器](https://developers.openai.com/codex/app/browser)使 Codex 能够检查渲染好的页面，操控它，并直接在正被审查的界面上响应批注。对页面或制品的评论始终紧紧贴合在工作闭环内，绝不会变成支离破碎的交接任务。

网页不仅是输出结果，更是控制面板。Codex 可以构建一个制品，在侧边栏打开它，细细检查、排查漏洞，并在原地将其打磨至完美。

以下界面的使用体验尤其出色：

- 用于轻量级静态界面的 index.html
- 用于 UI 审查的 Storybook
- 用于程序化动画的 Remotion Studio
- 基于浏览器的演示幻灯片
- 用于数据分析工作流的数据应用

区区一个 index.html 文件，无需服务器加持，就能蜕变为持久的交互式制品。线程自动化还会随着时间的推移不断刷新静态内容，保证你一回来，就能瞧见新鲜出炉的惊喜。

## **共享记忆**

当长线运行的线程能够跳出单一对话的窠臼，开始共享记忆时，它的实用价值便会翻倍。

> **共享记忆：** 存储在单一线程之外的持久化上下文。有了它，未来的工作就能基于清晰、可供审阅的信息顺畅接续。

一个稳妥的做法，是将持久化线程锚定在 Obsidian 库中。说白了，就是建一个存放纯文本文件的文件夹，确保它在漫长岁月里始终便于查阅、编辑和搬运。团队完全可以把这个文件夹扔进云存储、Git、Dropbox、Google Drive 或是任何顺手的文件同步工具里。

库的结构大致如下：

```
vault/
├── TODO.md
├── people/
├── projects/
├── agent/
└── notes/
```

在顶层目录，AGENTS.md 可以立下规矩：随着对团队人员、项目、决策及未闭环问题的了解日益加深，Codex 该如何去更新这个工作区。

没必要生搬硬套某个具体的库结构。你要做的是教会智能体去哪里寻找持久化上下文，弄清哪些上下文值得留存，并告诉它何时别瞎折腾。

比如，一份真正实用的 AGENTS.md 完全可以这样写：

- 把 ~/vault 视作持久化的工作记忆库。
- 优先使用规范统一的笔记，别让笔记碎片到处疯长。
- 给待办事项、人物、项目、每日小结和草稿笔记安排明确的归宿。
- 妥善保留关键决策、卡点阻碍、负责人、日期以及有价值的链接。
- 要是没什么实质性改动，就别去动库里的文件。

代码库用来装代码；而记忆库装的则是滚动翻新的上下文：涉及哪些人、改了什么、卡在哪里、有什么需要跟进，以及那些如果没记下来就会在两次会话间烟消云散的琐碎细节。

至关重要的上下文决不能只躺在某次对话的聊天记录里。得找个地方踏踏实实记下来，好让下一个线程能够无缝接棒。

Codex 也自带官方的[记忆功能](https://developers.openai.com/codex/memories)（位于 设置 > 个性化 > 记忆）。它们为用户的偏好设定、日常工作流以及踩过的坑提供了一个本地召回层。但这仅仅是对显性文字上下文的补充，绝非替代品。[Chronicle](https://developers.openai.com/codex/memories/chronicle) 功能亦是异曲同工，它能帮 Codex 根据你近期屏幕上闪过的内容快速构建记忆。

## **从代码向外破局**

Codex 的根基自然还是代码。但如今，围绕代码展开的周边工作，统统能在这个统一的体系内搞定：MCP 服务器、浏览器界面、桌面操控、线程自动化以及那些随时可查阅的生成制品。

这彻底颠覆了传统的控制模式。干预操作打断当前进程。排队机制把控后续任务。线程自动化保证人走茶不凉。而目标驱动则赋予了 Codex 一条清晰可见、值得全力冲刺的终点线。

时至今日，从下达指令、落地执行，再到制品的最终审阅，Codex 已能包揽整个工作流。即便战线拉出了代码库的边界，它依然能够游刃有余。
